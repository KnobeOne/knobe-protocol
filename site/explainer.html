Segment 1 of 4 — the problem Context survivorship bias

The fragment that survives transit is mistaken for the full knowledge object.

  • A file becomes a person
  • A credential becomes competence
  • A ranking becomes quality
  • A summary becomes the source
  • A model output becomes a decision basis
Compressed objects are treated as operationally sufficient.
The problem is not compression. It is compression without portable interpretive obligation.
Segment 1 of 4 — the problem The answer
KNOBE makes it harder for fragments
to pass as whole objects.
KNOBE preserves objecthood under compression.
Segment 2 of 4 — the protocol One file, three layers

Every .knobe.md file is simultaneously human-readable, machine-legible, and hash-verifiable.

layer 1 — yaml frontmatter
title, author, license, date — readable in any text editor in 2 seconds
layer 2 — markdown body
the document itself — opens in any editor, AI assistant, or browser
layer 3 — base64 json payload
attribution · transformation history · fidelity limits · SHA-256 integrity hash
A human reads it. A machine decodes it. A parser verifies it.
No conversion. No sidecar. No platform.
Segment 2 of 4 — the protocol Two protocol rules
integrity, not truth

The hash proves the record is unaltered since sealing. It does not prove accuracy, attribution honesty, or trustworthiness.

The green check is where inspection begins, not where it ends.

quarantine-first

Every new or external KNOBE defaults to quarantine until a human or governed system marks it trusted.

Inspection precedes action.

Attribution travels with the artifact. When the file circulates, the attribution circulates with it.
Segment 2 of 4 — the protocol Institutional positioning
A system of context,
not a system of record.
  • The LMS keeps grades
  • The IRB keeps approvals
  • The repository keeps deposits
  • KNOBE carries context between them
It does not replace those systems. It lowers the coordination cost across them.

Segment 4 of 4 — the invitation Two complementary layers

KNOBE is what the harness handles.

harness infrastructure

Controls the conditions around the agent — execution policy, tool permissions, observability, memory architecture.

The harness controls the encounter.

KNOBE — object layer

The artifact side of the agent harness vision — attribution, transformation history, fidelity limits, interpretive obligations carried by the object through any system.

The KNOBE preserves the object across encounters.

Complementary, not competing. If the artifact arrives stripped, no harness can reconstruct what was never carried.
